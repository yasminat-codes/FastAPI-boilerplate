# Runbook: Migration Failures

This runbook covers failures during Alembic database migrations: failed upgrades, schema drift, rollback decisions, and recovery paths. For background on the template's migration system, see the [Database Migrations](../database/migrations.md) guide.

## Symptoms

- `uv run db-migrate upgrade head` exits with a non-zero status or an unhandled exception.
- `uv run db-migrate-verify` reports schema drift (model state does not match the applied migrations).
- Application startup fails because the database schema does not match what the code expects.
- CI migration checks fail on a pull request.
- Users or operators report errors that trace back to missing columns, tables, or constraints.

## Types of Migration Failures

### Upgrade Failure During Deployment

The migration command ran during deployment but failed partway through.

**Diagnosis:**

1. Read the full error output from the migration command. Common causes:
   - **Syntax error in the migration script** — the autogenerated upgrade function has invalid SQL.
   - **Constraint violation** — the migration tries to add a NOT NULL column without a default to a table that already has rows.
   - **Lock timeout** — the migration is waiting for an exclusive table lock that another transaction holds.
   - **Connection failure** — the migration cannot connect to the database.

2. Check which revision the database is currently at:
   ```bash
   uv run db-migrate current
   ```

3. Check if the failed migration left the database in a partially applied state:
   ```bash
   uv run db-migrate history --verbose
   ```

**Resolution:**

The template's [migration guide](../database/migrations.md) documents three recovery paths. Choose based on the situation:

1. **Forward-fix** (preferred when possible):
   - Fix the migration script or add a follow-up migration that corrects the issue.
   - Run `uv run db-migrate upgrade head` again.
   - This is the safest path because it moves forward rather than trying to undo applied changes.

2. **Downgrade** (when the migration has a clean reverse path):
   ```bash
   uv run db-migrate downgrade -1
   ```
   - Only use this if the migration's `downgrade()` function is implemented and tested.
   - After downgrading, fix the migration script and upgrade again.

3. **Backup restore** (last resort):
   - Restore the database from the pre-deployment backup.
   - Fix the migration and redeploy.
   - This is only necessary when the migration left the database in an inconsistent state that neither forward-fix nor downgrade can resolve.

### Schema Drift

The database schema does not match what the Alembic migration chain produces when applied to an empty database. This means someone changed the schema outside of Alembic (manual DDL, ORM `create_all()`, or a skipped migration).

**Diagnosis:**

1. Run the drift detection command:
   ```bash
   uv run db-migrate-verify
   ```

2. If drift is detected, compare the live schema with the expected schema:
   ```bash
   uv run db-migrate revision --autogenerate -m "drift check"
   ```
   - This generates a new migration that represents the difference between the model definitions and the live database.
   - Review the generated migration to understand what drifted.
   - **Do not blindly apply it** — understand why the drift exists first.

**Resolution:**

1. If the drift is from a manual change that should be captured in a migration: keep the autogenerated revision, clean it up, and apply it.
2. If the drift is from `create_all()` or a development shortcut: the template's startup model does not call `create_all()`. If this happened, check for non-template code that creates tables at startup. Remove it and align with the migrations-only model.
3. If the drift is from a missing migration: write the migration, test it against a clean database, and apply it.

### Lock Timeout During Migration

Long-running migrations on large tables can be blocked by other transactions holding row or table locks, or the migration itself can block application queries while waiting for an exclusive lock.

**Diagnosis:**

1. Check for long-running transactions in PostgreSQL:
   ```sql
   SELECT pid, now() - pg_stat_activity.query_start AS duration, query, state
   FROM pg_stat_activity
   WHERE (now() - pg_stat_activity.query_start) > interval '5 minutes'
   ORDER BY duration DESC;
   ```

2. Check for lock contention:
   ```sql
   SELECT blocked_locks.pid AS blocked_pid,
          blocking_locks.pid AS blocking_pid,
          blocked_activity.query AS blocked_query,
          blocking_activity.query AS blocking_query
   FROM pg_catalog.pg_locks blocked_locks
   JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_activity.pid = blocked_locks.pid
   JOIN pg_catalog.pg_locks blocking_locks
     ON blocking_locks.locktype = blocked_locks.locktype
     AND blocking_locks.database IS NOT DISTINCT FROM blocked_locks.database
     AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation
     AND blocking_locks.page IS NOT DISTINCT FROM blocked_locks.page
     AND blocking_locks.tuple IS NOT DISTINCT FROM blocked_locks.tuple
     AND blocking_locks.virtualxid IS NOT DISTINCT FROM blocked_locks.virtualxid
     AND blocking_locks.transactionid IS NOT DISTINCT FROM blocked_locks.transactionid
     AND blocking_locks.classid IS NOT DISTINCT FROM blocked_locks.classid
     AND blocking_locks.objid IS NOT DISTINCT FROM blocked_locks.objid
     AND blocking_locks.objsubid IS NOT DISTINCT FROM blocked_locks.objsubid
     AND blocking_locks.pid != blocked_locks.pid
   JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_activity.pid = blocking_locks.pid
   WHERE NOT blocked_locks.granted;
   ```

**Resolution:**

1. If a long-running query is blocking the migration: decide whether to wait for it to finish or terminate it with `pg_terminate_backend(pid)`.
2. For migrations on large tables, use the expand-contract pattern documented in the [Migrations guide](../database/migrations.md):
   - Phase 1 (expand): add the new column as nullable with no default, no lock required.
   - Phase 2 (backfill): populate the column in batches using a script.
   - Phase 3 (contract): add the NOT NULL constraint or default once all rows are populated.
3. Run heavy migrations during a maintenance window when application load is low.

### CI Migration Check Failure

The GitHub Actions migration check workflow (`migration-check.yml`) runs `uv run db-migrate upgrade head` and then `uv run db-migrate-verify` against a clean PostgreSQL container. If it fails, the pull request has a migration problem.

**Diagnosis:**

1. Read the CI output for the specific error.
2. Common causes:
   - The new migration depends on a revision that does not exist (wrong `down_revision`).
   - The migration chain has multiple heads (two migrations point at the same parent).
   - The autogenerated migration includes operations that are not compatible with the current model state.

**Resolution:**

1. **Multiple heads**: linearize the chain by setting the correct `down_revision` in one of the conflicting migrations:
   ```bash
   uv run db-migrate heads  # shows all heads
   uv run db-migrate merge heads -m "merge migration branches"  # or manually fix down_revision
   ```

2. **Missing revision**: confirm that all migration files are committed and that the `down_revision` references an existing revision ID.

3. **Model mismatch**: regenerate the migration against a fresh database and compare with the committed version.

## Prevention

1. **Always test migrations against a clean database** before deploying. The template provides `uv run db-migrate-verify` for this purpose.
2. **Never modify the database schema outside of Alembic** in environments where migration tracking matters (staging, production).
3. **Use expand-contract for destructive changes**. Never drop columns, rename tables, or add NOT NULL constraints in a single migration on tables with existing data.
4. **Take a backup before running migrations in production**. The backup is your last-resort recovery path.
5. **Run migrations as a separate deployment step**, not during application startup. The template enforces this by not calling `create_all()` at boot.
6. **Include migration verification in CI**. The template already provides a GitHub Actions workflow for this.
